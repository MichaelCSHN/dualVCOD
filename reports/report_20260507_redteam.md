## 🔴 mIoU 0.8705 INVALID (DATA LEAKAGE DETECTED) 🔴

**The claimed mIoU=0.8705 is scientifically INVALID as an out-of-sample
generalization metric.** All 28 MoCA validation videos (1,188 temporal windows)
were present in the training `ConcatDataset` and received gradient updates
during training. See Task 1 for full forensic analysis.

---

## Executive Summary

A **critical data leakage bug** was discovered in `tools/train.py`. The `ConcatDataset`
used for training includes the **FULL MoCA dataset** (all 141 videos / 6,948 windows),
while the validation set is a 20% video-level split of the SAME dataset. This means
**all 28 validation videos received gradient updates during training**, completely
invalidating the claimed mIoU of 0.8705 as a measure of out-of-sample generalization.

Additionally, hard baselines reveal that the model's temporal module is **completely
order-invariant** (forward = reversed = 0.8706), and that **GT propagation alone
outperforms the model** (0.8734 > 0.8706), indicating that objects barely move
between consecutive annotated frames.

**The model must be retrained with the data leak fixed before any publication.**

## Task 1: Data Leakage Death Audit

### 1.1 Root Cause

**File**: `tools/train.py`, lines 165-197

The critical bug is on **lines 165-174**:

```python
# Line 165-174: Build joint training set
train_sets = []
for root in DATASET_ROOTS:
    if os.path.isdir(root):
        ds = RealVideoBBoxDataset([root], T=args.T, target_size=224, augment=True)
        train_sets.append(ds)                           # ← MoCA: ALL 141 videos!

joint_train_ds = ConcatDataset(train_sets)              # ← VAL VIDEOS INCLUDED!
```

Then on **lines 180-197**, the `split_by_video()` function is called, producing
`train_idx` and `val_idx`. However, `train_idx` is **NEVER used** to filter the
training data. Only `val_idx` is consumed — to create the validation DataLoader:

```python
# Line 182-183: split_by_video produces train_idx, val_idx
train_idx, val_idx = split_by_video(moca_ds, val_ratio=0.2, seed=42)

# Line 192-195: train_loader uses joint_train_ds (FULL MoCA, not filtered!)
train_loader = DataLoader(
    joint_train_ds, ...)

# Line 196-199: val_loader correctly uses val_idx
val_loader = DataLoader(
    Subset(moca_ds, val_idx), ...)
```

**The `train_idx` exists but is completely unused.** It was generated by
`split_by_video()` but never applied to filter the MoCA portion of
`ConcatDataset`.

### 1.5 Fix

The fix is a **one-line change** in `tools/train.py`:

```python
# BEFORE (BUGGY) — line 165-174:
train_sets = []
for root in DATASET_ROOTS:
    if os.path.isdir(root):
        ds = RealVideoBBoxDataset([root], T=args.T, target_size=224, augment=True)
        train_sets.append(ds)                    # ← MoCA: ALL videos
joint_train_ds = ConcatDataset(train_sets)

# AFTER (FIXED):
train_sets = []
for root in DATASET_ROOTS:
    if os.path.isdir(root):
        ds = RealVideoBBoxDataset([root], T=args.T, target_size=224, augment=True)
        if "MoCA" in root and not "MoCA_Mask" in root:
            # Filter MoCA to train split only
            ds = Subset(ds, train_idx)           # ← Only train videos
        train_sets.append(ds)
joint_train_ds = ConcatDataset(train_sets)
```

Note: The `split_by_video()` call must be moved BEFORE the `train_sets` loop.

### 1.6 Impact on Previous Audits

The Phase 2.1 Compliance Audit (v1 and v2 refined) **did NOT detect** this leak
because Task 1 (Physical Isolation Audit) only checked whether the 28 MoCA val
video IDs overlapped with MoCA_Mask and CAD video IDs — it did NOT check whether
the MoCA val videos were in the MoCA training set itself.

The compliance audit compared `train_videos` (from `split_by_video`'s `train_idx`)
against the val videos. But the actual training was done on the FULL MoCA dataset,
not on `train_idx`. The audit framework trusted that `train_idx` was used for
training — it was not.

**This is a failure of the compliance audit's scope, not its methodology.** The
audit correctly checked cross-dataset leakage (MoCA vs MoCA_Mask vs CAD), which
was clean. It did not check intra-dataset leakage (MoCA train vs MoCA val within
the same dataset).

## Task 2: Hard Baselines Challenge

All baselines evaluated on the MoCA validation set (1,188 clips, 28 videos).

### 2.1 Results Summary

| # | Baseline | mIoU | Recall@0.5 | Δ vs Forward |
|---|----------|------|------------|-------------|
| B1 | **GT Propagation (prev-frame oracle)** | **0.8734** | 0.9643 | **+0.0028** ← BETTER than model! |
| B4 | Normal Forward T=5 (model) | 0.8706 | 0.9340 | — |
| B2 | Temporal REVERSED [f4..f0] | 0.8706 | 0.9340 | 0.0000 |
| B3 | T=1 approx (same frame x5) | 0.7753 | 0.8769 | −0.0953 |

### 2.2 Analysis

**B1 — GT Propagation (0.8734 > 0.8706)**:

This is the most damning finding. Simply carrying forward the **previous frame's
ground-truth bounding box** achieves HIGHER mIoU than the trained model. This
means:
- Objects in MoCA barely move between consecutive annotated frames (5-frame
  intervals at the dataset's annotation stride)
- The model's 0.8706 mIoU is largely explained by frame-to-frame temporal
  smoothness, not by genuine camouflage-breaking capability
- The model is effectively learning to predict "whatever the previous frame's
  box was" — which is a valid strategy when objects don't move, but is not
  camouflage breaking

**B2 — Temporal REVERSED (0.8706 = 0.8706)**:

The reversed-order result is **identical** to the forward result. This proves
that the TemporalNeighborhood module is **completely order-invariant** — it
processes T frames as an unordered set, extracting spatial features without
respect to temporal direction. The "temporal" module is functionally a
**frame-pooling** module, not a true temporal dynamics module.

This is consistent with the module's architecture: `GlobalAvgPool` over the
temporal dimension eliminates order information. Only the `Conv1d(k=3)` could
capture order, but the subsequent global pooling destroys it.

**B3 — T=1 (0.7753)**:

Removing inter-frame variation causes a 0.0953 drop, confirming that multi-frame
information provides benefit. However, given B2's finding of order-invariance,
this benefit comes from seeing the object/scene from multiple slightly different
viewpoints (spatial augmentation through time), not from modeling motion dynamics.

### 2.3 Verdict on Temporal Module

| Question | Answer |
|----------|--------|
| Does multi-frame input help? | Yes (+0.095 over T=1) |
| Is temporal ORDER used? | **No** — forward ≡ reversed |
| Does model beat GT propagation? | **No** — 0.8706 < 0.8734 |
| Is the module "temporal"? | No — it's a frame-pooling module |

The `TemporalNeighborhood` module should be renamed to `MultiFramePooling` to
accurately reflect its behavior. True temporal modeling would require:
- Causal masking (future frames cannot influence past predictions)
- Positional encoding (so order matters)
- Directional convolutions (not symmetric padding)

## Task 4: Final 7-Point Verdict

### Point 1: Is the current mIoU=0.8705 trustworthy (leakage-free)?

**NO. The mIoU=0.8705 is INVALID as an out-of-sample generalization metric.**

The model was evaluated on 28 MoCA videos (1,188 clips) that were ALL present
in the training `ConcatDataset`. Every validation window received gradient
updates during training. The 0.8705 mIoU reflects **memorization of seen data**,
not generalization to unseen videos.

Additionally, the model underperforms a trivial GT-propagation baseline
(0.8706 < 0.8734), indicating that most of its "performance" comes from
temporal smoothness rather than camouflage breaking.

### Point 2: Is there data leakage? (specific code location)

**YES.** The leak is in `tools/train.py`, lines 165-174:

```python
# Line 168-169: MoCA dataset loaded with ALL 141 videos
ds = RealVideoBBoxDataset([root], T=args.T, target_size=224, augment=True)
train_sets.append(ds)
# Line 174: ConcatDataset includes the 28 val videos
joint_train_ds = ConcatDataset(train_sets)
```

The `split_by_video()` at line 182 correctly produces `train_idx` (indices for
113 videos, 5,760 windows) and `val_idx` (indices for 28 videos, 1,188 windows).
However, `train_idx` is **never used** to filter the training data. It is
generated but immediately discarded.

**Root cause**: `split_by_video()` was called, but its `train_idx` output was
never wired into the training DataLoader construction.

### Point 3: Is the temporal module genuinely effective?

**PARTIALLY. Multi-frame input helps (+0.095 over T=1), but the module is
completely order-invariant (forward ≡ reversed = 0.8706).**

The module is more accurately described as a "multi-frame pooling" module rather
than a "temporal" module. It aggregates spatial features across T frames without
respecting temporal order. True temporal dynamics (causality, direction, motion)
are not modeled.

Furthermore, the module's benefit is largely explained by the fact that objects
barely move between frames — the GT propagation baseline (0.8734) outperforms
the full model (0.8706).

**Recommendation**: Redesign the temporal module for Phase 3:
- Add causal masking (no future-frame peeking)
- Add temporal positional encoding
- Replace symmetric Conv1d with causal Conv1d
- Consider replacing GlobalAvgPool with an order-sensitive aggregation

### Point 4: What is the strict boundary with GreenVCOD (arXiv:2501.10914)?

**We share only the high-level "temporal window" concept. Zero lines of code
are reused. The architectures, training paradigms, and implementations are
completely different.**

| Boundary | Status |
|----------|--------|
| Concept ("temporal window for VCOD") | Shared (but not novel to GreenVCOD) |
| Backbone architecture | Different (MobileNetV3 vs PixelHop) |
| Training paradigm | Different (gradient descent vs Green Learning) |
| Loss function | Different (SmoothL1+GIoU vs N/A) |
| Code reuse | **0 lines** |
| Data pipeline | Independently built |
| Temporal module design | Independently designed |

We should cite GreenVCOD as related work that also uses temporal windows for
VCOD, but clearly position our work as an independent, gradient-based
lightweight approach.

### Point 5: Does this work have independent publication value?

**YES — but only after fixing the data leak and retraining.**

The architecture has genuine merits:
- Extreme lightweight design (<1.5M params, >100 FPS)
- Independent PyTorch implementation
- Strong cross-species generalization (CAD ablation)
- Methodological rigor (compliance audit framework)

However, the current mIoU=0.8705 **cannot be used** in any publication. The
retrained model's honest mIoU (on truly unseen data) will likely be lower,
but honest.

**Target venue**: CVPR/ICCV workshop, or mid-tier conference (BMVC, WACV).

### Point 6: What is the confirmed method name?

**`MicroVCOD`** — Swift Video Camouflaged Object Detection.

`MicroVCOD` is the alternative if emphasizing the <2M parameter count.

`MicroVCOD` is recommended because:
1. The 118.2 FPS real-time speed is the headline differentiator
2. "Swift" is more memorable and marketable
3. It avoids confusion with "Micro" (which sounds like a smaller version of
   something else)

### Point 7: What is the next step?

**FIX THE DATA LEAK → RETRAIN → THEN WRITE THE PAPER.**

Immediate action items:

   - Move `split_by_video()` before `train_sets` loop

4. **Update the compliance audit** to check intra-dataset leakage

5. **Proceed to Phase 3 (IR dual-modality)** on the clean foundation

**Do NOT submit or share the current mIoU=0.8705 as a result.** It is
