## 1. Core Hypothesis

Foreground-background ambiguity (confirmed in E-41.5 for flatfish_2 and white_tailed_ptarmigan) can be reduced by replacing backgrounds during training. The intuition: if the model sees a camouflaged object composited onto diverse backgrounds, it learns to identify the object by its own features rather than relying on background context.

## 2. Background Mixing Implementation

### 2.1 Modifications to `src/dataset_real.py`

**New constructor parameter**: `bg_mix_prob=0.0` — probability per sample of applying background replacement.

**Three new methods**:

- `_load_full_mask(sample, frame_idx, bbox_norm)` — loads binary mask at `target_size` (224×224):
  - MoCA_Mask/CAD: reads PNG ground truth mask via `_resolve_mask_path()`, resizes to 224×224 with `cv2.INTER_AREA`
  - MoCA CSV (bbox-only): generates rectangle mask from bbox via `_bbox_to_mask(bbox_norm, 224, 224)`

- `_pick_background(exclude_dir)` — selects background frame from different video:
  - Randomly samples from `self.samples` (up to 50 attempts)
  - Excludes samples from the same `frame_dir` to avoid self-compositing
  - Loads first frame of the sampled clip via `_load_or_decode()`
  - Returns float tensor (3, 224, 224) in [0, 1]

- `_composite_frame(frame, mask, background)` — static method, hard binary compositing:
  ```python
  mask_t = torch.from_numpy(mask).float().unsqueeze(0)  # (1, H, W)
  return frame * mask_t + background * (1.0 - mask_t)
  ```
  No Gaussian blur, no edge softening — direct binary cutout per user instruction to avoid over-engineering fake samples.

### 2.2 Integration in `__getitem__`

Trigger logic (before per-frame loop):
```python
do_bg_mix = (
    self.augment and self.bg_mix_prob > 0
    and random.random() < self.bg_mix_prob
)
```

When triggered:
1. **One background** selected for ALL T frames (temporal consistency)
2. For each frame in clip:
   - Load frame → convert to float tensor
   - Load full mask via `_load_full_mask()`
   - If mask has both foreground AND background pixels: composite
   - If mask is all-0 or all-1: skip compositing
   - Apply flip and jitter AFTER compositing

### 2.3 Configuration

```json
{ "bg_mix_prob": 0.5 }
```

All other parameters identical to E-39: backbone=mobilenet_v3_small, head=dense_fg_aux, lr=0.001, bs=16, T=5, T_max=8, jitter_strength=0.15, dense_fg weight=0.5.

## 3. True Mask vs Bbox-Only Sample Handling

| Sample type | Train % | Mask source for compositing | Mask source for dense supervision |
|-------------|---------|----------------------------|-----------------------------------|
| MoCA_Mask | ~34% | PNG GT mask (224×224) | PNG GT mask (28×28) |
| CAD | ~2% | PNG GT mask (224×224) | PNG GT mask (28×28) |
| MoCA CSV | ~64% | Bbox rectangle (224×224) | Bbox rectangle (28×28) |

Both compositing and dense supervision use the **same mask source** per sample type. For bbox-only samples, both fall back to `_bbox_to_mask()` — a rectangular mask derived from the GT bbox. This means the composited training sample for bbox-only data has a rectangular cutout, which is a stronger domain shift than mask-based compositing.

## 4. Bbox and Dense Target Synchronization — CONFIRMED

The implementation preserves correctness of both supervision targets:

1. **Compositing happens before flip** — the full mask is loaded with `bbox_before_flip`, so the mask aligns with the pre-flip image
2. **Flip is applied to image, bbox, AND dense mask together** — after compositing, the same `do_flip` flag flips the image tensor (`torch.flip`), the bbox coordinates (`1.0 - bbox[2]`, `1.0 - bbox[0]`), and the 28×28 dense mask (`np.fliplr`)
3. **Compositing does NOT change GT bbox** — foreground position is preserved, so the original GT bbox remains correct
4. **Dense mask loaded AFTER compositing** — the 28×28 mask for BCE loss is loaded from the (possibly flipped) bbox, ensuring it matches the final training target

**Conclusion**: Both bbox regression target and dense foreground BCE target are correctly synchronized with the composited image. The failure is NOT caused by label misalignment.

## 7. Did Background Mixing Reduce Foreground-Background Ambiguity?

**No. It made it catastrophically worse.**

Root cause analysis:

1. **Domain shift from compositing artifacts**: The hard binary cutout creates sharp boundaries between foreground and background that don't exist in real camouflaged images. In real camouflage, the transition between object and background is gradual and defined by subtle texture/color differences. The composited training samples have an artificial high-contrast edge that the model latches onto as a shortcut.

2. **Model learns compositing boundary, not object features**: Instead of learning "what does a flatfish look like," the model learns "where is the sharp rectangular boundary in this image." On real validation images (no compositing boundaries), this cue is absent, and the model fails.

3. **Background diversity backfires**: The model sees the same foreground object on many different backgrounds. Rather than learning the invariant features of the object, it learns that "anything with a sharp edge could be foreground" — leading to massive overprediction (area_ratio=4.0, pred_too_large=2101).

4. **Bbox-only samples have rectangular artifacts**: For 64% of training data (MoCA CSV), the "foreground" is a perfect rectangle. The model learns to detect rectangles, not camouflaged animals.

5. **Compositing breaks camouflage**: In real images, the camouflage IS the challenge — the object looks like the background. By replacing the background, we remove the very thing the model needs to learn to overcome. The training distribution no longer contains the hard cases that the validation set tests.

## 8. Should We Pursue 30ep?

**NO. Strong recommendation against.**

Reasons:
1. Training trajectory shows saturation by epoch 3 with zero val improvement from epoch 3→8 (0.2733 → 0.2777)
2. Train-val gap widens monotonically (0.12 → 0.47) — more epochs = more overfitting, NOT convergence
3. 8ep is sufficient to show direction validity for dense_fg_aux (E-39 reached 96.6% of E-40 ceiling)
4. The fundamental problem is NOT insufficient training — it's a domain shift that training AMPLIFIES with more epochs
5. GPU time is better spent on directions that address the root cause

**Do not iterate on background mixing parameters** (bg_mix_prob, edge softening, blending alpha, etc.). The core mechanism — replacing natural backgrounds with synthetic composites — is fundamentally broken for camouflage detection.
